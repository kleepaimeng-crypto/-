# 数据模拟器开发指导

> 目标：开发一个本地数据模拟器，按 `接口格式Schema.md` 中的数据结构，通过 UDP 协议向本地端口持续发送模拟数据。  
> 范围：重点保证 `qar.frame`、`smart_window.status`、`ife_633.behavior` 三类接口的数据合理性；其他接口可先用固定或低频变化数据填充。  
> 约束：模拟中只出现 1 架飞机，所有接口共享同一航班、任务、航线、时间和乘客上下文。

## 1. 总体功能

模拟器启动后生成一组全局场景上下文：

| 上下文项 | 说明 |
| --- | --- |
| `taskId` | 本次模拟任务唯一标识，所有相关接口共用 |
| `flightNumber` | 航班号，QAR 使用 `FLIGHT NUMBER`，其他接口按各自字段映射 |
| `originAirport` | 起飞机场，QAR 使用 `ORIGIN` |
| `destinationAirport` | 到达机场，QAR 使用 `DESTINATION` |
| `scenarioStartTime` | 场景开始时间 |
| `simulatedNow` | 当前模拟时间 |
| `aircraftModel` | 宽体客机机型 |
| `passengerCount` | 本次模拟乘客数量 |
| `windowRows` | 舷窗排数 |
| `seatMap` | 乘客、座位、终端、舷窗关联关系 |

所有 UDP 发送器使用同一个上下文，不允许各接口自行随机生成航班号、任务 ID、起降点、乘客或时间。

## 2. UDP 端口和发送周期配置

### 2.1 端口宏变量

所有接口端口号在全局宏变量中定义。默认第一个端口为 `8090`，后续按顺序递增。

```python
UDP_HOST = "127.0.0.1"

PORT_QAR_FRAME = 8090
PORT_GROUND_TASK = 8091
PORT_GROUND_TRAFFIC_RECORD = 8092
PORT_GROUND_SESSION_SUMMARY = 8093
PORT_SMART_WINDOW_STATUS = 8094
PORT_IFE_633_BEHAVIOR = 8095
PORT_IFE_COCKRELL_BEHAVIOR = 8096
```

### 2.2 发送周期宏变量

默认发送间隔为 10 秒。每个接口必须支持单独配置发送周期。

```python
DEFAULT_SEND_INTERVAL_SECONDS = 10

SEND_INTERVAL_QAR_FRAME_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
SEND_INTERVAL_GROUND_TASK_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
SEND_INTERVAL_GROUND_TRAFFIC_RECORD_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
SEND_INTERVAL_GROUND_SESSION_SUMMARY_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
SEND_INTERVAL_SMART_WINDOW_STATUS_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
SEND_INTERVAL_IFE_633_BEHAVIOR_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
SEND_INTERVAL_IFE_COCKRELL_BEHAVIOR_SECONDS = DEFAULT_SEND_INTERVAL_SECONDS
```

### 2.3 推荐配置文件结构

宏变量提供默认值，运行时允许从配置文件覆盖。

```json
{
  "udpHost": "127.0.0.1",
  "ports": {
    "qar.frame": 8090,
    "ground.task": 8091,
    "ground.traffic_record": 8092,
    "ground.session_summary": 8093,
    "smart_window.status": 8094,
    "ife_633.behavior": 8095,
    "ife_cockrell.behavior": 8096
  },
  "sendIntervalsSeconds": {
    "default": 10,
    "qar.frame": 10,
    "ground.task": 10,
    "ground.traffic_record": 10,
    "ground.session_summary": 10,
    "smart_window.status": 10,
    "ife_633.behavior": 10,
    "ife_cockrell.behavior": 10
  }
}
```

## 3. 统一航班上下文规则

### 3.1 中国 5 个城市机场

每次启动从下表随机选择两个不同机场作为起点和终点。

| 机场代码 | 城市 | 机场 | 纬度 | 经度 |
| --- | --- | --- | ---: | ---: |
| `ZBAA` | 北京 | 北京首都国际机场 | 40.0799 | 116.6031 |
| `ZSPD` | 上海 | 上海浦东国际机场 | 31.1443 | 121.8083 |
| `ZGGG` | 广州 | 广州白云国际机场 | 23.3924 | 113.2988 |
| `ZUUU` | 成都 | 成都双流国际机场 | 30.5785 | 103.9471 |
| `ZSHC` | 杭州 | 杭州萧山国际机场 | 30.2369 | 120.4324 |

### 3.2 飞行阶段

模拟飞行中段，即巡航阶段。

| 字段 | 规则 |
| --- | --- |
| 当前位置 | 位于起点到终点大圆航线的 35% 到 65% 之间 |
| 高度 | 约 31000 到 39000 英尺 |
| 地速 | 约 430 到 510 节 |
| 空速 | 约 250 到 330 节，或按文档字段填近似值 |
| 航向角 | 接近起点到终点的方位角，允许小幅波动 |
| 俯仰角 | 巡航阶段小幅正值，建议 -1.0 到 3.0 度 |
| 横滚角 | 巡航阶段小幅摆动，建议 -5.0 到 5.0 度 |
| 地面状态 | `AIR GND ON GND = AIR` 或等价非地面状态 |
| 剩余航程 | 随模拟时间逐步减少 |

### 3.3 字段映射

| 统一上下文 | QAR 字段 | ground 字段 | IFE 字段 |
| --- | --- | --- | --- |
| `taskId` | 无原生字段，可不填 | `taskId` | 可放入扩展字段或内部状态，不破坏原 schema |
| `flightNumber` | `FLIGHT NUMBER` | `flightNo` | `sysInfo.flightId` |
| `originAirport.code` | `ORIGIN` | 场景名称中体现 | 无 |
| `destinationAirport.code` | `DESTINATION` | 场景名称中体现 | 无 |
| `simulatedNow` | `time` | `startedAt`、窗口时间 | `sysInfo.timestamp` |
| `latitude` | `PRES POSN LAT - FMC` | 无 | 无 |
| `longitude` | `PRES POSN LONG - FMC` | 无 | 无 |

时间格式按接口适配：

| 接口 | 时间格式 |
| --- | --- |
| QAR `time` | `HH:mm:ss` |
| ground 系列 | ISO 8601，例如 `2026-06-10T19:20:00+08:00` |
| 智能舷窗 | `yyyy-MM-dd HH:mm:ss.sss` |
| IFE | `yyyy-MM-dd HH:mm:ss.sss` |

## 4. 飞机和乘客模型

### 4.1 机型

建议固定为大型宽体客机：

| 项 | 建议值 |
| --- | --- |
| `aircraftModel` | `Airbus A330-200` |
| 座舱布局 | 公务舱 30 座 + 经济舱 207 座 |
| 模拟乘客数 | 237 |
| 舷窗位置数 | 58 |
| 舷窗编号范围 | `1~116`，左侧 1–58，右侧 59–116 |

### 4.2 乘客生成

按 A330-200 的 237 个有效座位生成乘客，不得按固定每排座位数推算。每个乘客包含：

| 字段 | 规则 |
| --- | --- |
| `userId` | `PAX-00001` 起递增 |
| `pnr` | 6 位订座编码，字母数字混合 |
| `seatNo` | 按 A330-200 座位分布生成，统一为字母加排号，例如 `A12` |
| `cabinClass` | `BUSINESS`、`ECONOMY` |
| `deviceId` | `SVDU-<row>-<seat>` 或 `ANDROID-xxxx` |
| `terminalId` | `T-0001` 起递增 |
| `displayTerminalId` | 可用座位号或展示编号 |

### 4.3 座位和舷窗关系

公务舱 11–15 排使用 `A C D H J L`；经济舱按确认布局使用 31–57 排及其特殊排座位。模拟器内部座位清单必须恰好 237 个且编号唯一。

舷窗状态不要求绑定到每名乘客。ID 1–58 为左侧，59–116 为右侧；左右侧使用相同的区域划分：

| 区域 | `zoneId` | 每侧序号 |
| --- | ---: | --- |
| 前舱 | 1 | 1–17 |
| 中舱 | 2 | 18–37 |
| 后舱 | 3 | 38–58 |

## 5. 重点接口生成规则

### 5.1 `qar.frame`

`qar.frame` 是飞行状态的关键模拟接口，必须保证飞行数据连续、合理。

### 5.1.1 生成规则

| 字段 | 规则 |
| --- | --- |
| `FLIGHT NUMBER` | 使用全局 `flightNumber` |
| `ORIGIN` | 使用全局 `originAirport.code` |
| `DESTINATION` | 使用全局 `destinationAirport.code` |
| `PRES POSN LAT - FMC` | 从航线中点附近逐步推进，保留 6 到 9 位小数 |
| `PRES POSN LONG - FMC` | 从航线中点附近逐步推进，保留 6 到 9 位小数 |
| `BARO COR ALT NO. 1` | 31000 到 39000，缓慢波动 |
| `GROUNDSPEED` | 430 到 510，缓慢波动 |
| `COMPUTED AIRSPEED` | 250 到 330，缓慢波动 |
| `TRACK ANGLE TRUE - FMC` | 接近航线方位角，小幅波动 |
| `CAPT DISPLAY HEADING` | 接近 `TRACK ANGLE TRUE - FMC` |
| `BODY PITCH RATE` | -1.0 到 3.0 |
| `BODY ROLL RATE` | -5.0 到 5.0 |
| `LT MAIN FUEL QTY` | 随时间缓慢减少 |
| `RT MAIN FUEL QTY` | 随时间缓慢减少，与左油箱接近 |
| `CENTER MAIN FUEL QTY` | 随时间缓慢减少 |
| `AIR GND ON GND` | 巡航中固定为 `AIR` |
| `DISTANCE TO GO` | 根据航线进度递减 |
| `DESTINATION ETA` | 根据剩余距离和地速估算 |
| `frameCount` | 每次发送递增 |
| `time` | 使用当前模拟时间，格式 `HH:mm:ss` |

### 5.1.2 合理性要求

- 经纬度不能跳变，每次发送只允许沿航线小幅前进。
- 高度、速度、航向角不能完全随机，必须以当前值为基础加小幅扰动。
- 横滚角不能长期大幅偏转，巡航大多数时间接近 0。
- 左右主油箱油量差异应小，不能一边快速归零。

### 5.2 `smart_window.status`

`smart_window.status` 是舷窗状态关键模拟接口。每次发送要体现细节变化。

### 5.2.1 生成规则

| 字段 | 规则 |
| --- | --- |
| `windowId` | `1~116` |
| `zoneId` | 每侧序号 1–17 为前舱、18–37 为中舱、38–58 为后舱 |
| `brightnessLevel` | `0~10`，每次只变化 `-1`、`0` 或 `+1` |
| `connectStatus` | 大多数为 `true` 或 `1`，少量临时断连 |
| `status` | 大多数为 `NORMAL`，少量 `FAULT`，极少 `TEST` |
| `timestamp` | 使用全局模拟时间，格式 `yyyy-MM-dd HH:mm:ss.sss` |

### 5.2.2 变化策略

每次发送建议不要把所有舷窗全部重置。应保存上一轮舷窗状态，然后做增量更新：

| 变化项 | 建议 |
| --- | --- |
| 亮度 | 每个舷窗 30% 概率变化一级 |
| 断连 | 每轮约 1% 舷窗断连 |
| 故障 | 每轮约 0.5% 舷窗为 `FAULT` |
| 测试 | 每轮约 0.2% 舷窗为 `TEST` |
| 批量发送 | 可以一次发送全部舷窗数组，也可以每次发送部分变更 |

示例：某个舷窗上次 `brightnessLevel=6`，本次只能变为 `5`、`6` 或 `7`，不能突然跳到 `0` 或 `10`。

### 5.3 `ife_633.behavior`

`ife_633.behavior` 是乘客行为关键模拟接口，重点模拟每个用户的视频和音乐偏好。每个乘客应该长期保持相对稳定的偏好，但行为事件可动态变化。

### 5.3.1 视频类型

视频内容允许多类型组合。

```text
爱情、都市、青春、奇幻、武侠、古装、科幻、猎奇、竞技、传奇、逆袭
```

生成规则：

- 每名乘客随机分配 1 到 3 个视频偏好类型。
- `contentType` 可以使用单个类型，也可以用 `/` 拼接多类型，例如 `科幻/竞技`。
- 同一乘客后续电影行为优先从自己的偏好中选择。

### 5.3.2 音乐类型

音乐内容允许多类型组合。

```text
流行、摇滚、民谣、电子、舞曲、说唱、轻音乐、爵士、乡村、R&B、古典
```

注意：用户原始需求中写到“流型”，开发时建议按“流行”处理；如需严格保留原值，可配置别名映射。

生成规则：

- 每名乘客随机分配 1 到 3 个音乐偏好类型。
- `musicType` 可以使用单个类型，也可以用 `/` 拼接多类型，例如 `流行/R&B`。
- 同一乘客后续音乐行为优先从自己的偏好中选择。

### 5.3.3 行为类型分布

633 IFE 支持：

| `behaviorType` | 建议比例 | 说明 |
| --- | ---: | --- |
| `MOVIE_PLAY` | 45% | 电影播放 |
| `MUSIC_PLAY` | 35% | 音乐播放 |
| `CAST_SCREEN` | 5% | 头等舱投屏，仅头等舱乘客高概率出现 |
| `WAP_BROWSING` | 15% | WAP 上网 |

### 5.3.4 分页或分包要求

用户多时注意分页格式。由于当前模拟器通过 UDP 发送，建议把“分页”理解为分包：

| 字段 | 规则 |
| --- | --- |
| `page` | 当前分包页码，从 1 开始 |
| `pageSize` | 每包乘客行为数量，例如 50 |
| `total` | 本轮计划发送的行为总数 |
| `items` | 行为数组 |

推荐 UDP payload：

```json
{
  "messageType": "ife_633.behavior",
  "sentAt": "2026-06-10T19:42:20+08:00",
  "page": 1,
  "pageSize": 50,
  "total": 237,
  "items": []
}
```

`items` 中每项仍按 `接口格式Schema.md` 的 IFE 顶层结构：

```json
{
  "sysInfo": {},
  "paxInfo": {},
  "behaviorInfo": {},
  "extInfo": {}
}
```

## 6. 其他接口生成规则

### 6.1 `ground.task`

可使用固定数据，但必须和全局上下文一致。

| 字段 | 规则 |
| --- | --- |
| `taskId` | 全局 `taskId` |
| `flightNo` | 全局 `flightNumber` |
| `scenarioName` | `${origin.city} -> ${destination.city} 巡航模拟` |
| `status` | 默认 `running` |
| `phase` | 默认 `cruise` |
| `terminalCount` | 乘客数量或活跃终端数量 |
| `startedAt` | 全局 `scenarioStartTime` |
| `totalBytes` | 可随 traffic 累加，初期可固定递增 |

### 6.2 `ground.traffic_record`

可用固定规则生成，保持和乘客、终端一致：

- `taskId` 使用全局 `taskId`。
- `terminalId`、`displayTerminalId`、`seatLabel` 来自乘客座位表。
- `application` 可从 `高清视频`、`音乐`、`网页浏览` 中选择。
- `throughputMbps` 与应用类型相关，视频高、音乐低、网页中等。

### 6.3 `ground.session_summary`

可用固定或低频变化数据：

- `taskId` 使用全局 `taskId`。
- `terminalId`、`displayTerminalId`、`seatLabel` 来自乘客座位表。
- `application` 与乘客当前 IFE 行为关联。
- `durationSeconds` 随模拟时间递增。
- `status` 大多数为 `active`。

## 7. `ife_633.behavior` 和 `ife_cockrell.behavior` 统一规则

这两个接口内容基本类似，必须共用乘客、航班、偏好和行为生成基础逻辑。

### 7.1 共用部分

| 模块 | 统一要求 |
| --- | --- |
| `sysInfo` | 同一 `timestamp` 格式和 `flightId` |
| `paxInfo` | 使用同一乘客表 |
| 视频行为 | 共用视频偏好、播放动作、播放进度逻辑 |
| 音乐行为 | 共用音乐偏好、播放动作、播放进度、音量逻辑 |
| WAP 行为 | 共用会话、IP、协议、端口生成逻辑 |
| `extInfo` | 共用错误码和错误描述策略 |

### 7.2 差异部分

| 接口 | 差异 |
| --- | --- |
| `ife_633.behavior` | 支持 `CAST_SCREEN`，不支持 `SHOPPING` |
| `ife_cockrell.behavior` | 支持 `SHOPPING`，不支持 `CAST_SCREEN` |
| `ife_cockrell.behavior` | 电影和音乐可额外生成封面 Base64、媒体类型和校验和 |

### 7.3 实现建议

代码层建议先生成统一的内部事件对象：

```json
{
  "passenger": {},
  "behaviorKind": "MOVIE_PLAY",
  "mediaPreference": ["科幻", "竞技"],
  "timestamp": "2026-06-10 19:42:20.000"
}
```

然后分别转换成 633 IFE 和科克瑞尔 IFE 的接口格式。这样能保证两个接口数据统一，同时适配字段差异。

## 8. 推荐模块拆分

| 模块 | 职责 |
| --- | --- |
| `config` | 端口宏、发送周期、运行配置加载 |
| `scenario` | 生成单架飞机、航线、任务、时间上下文 |
| `flight_model` | 计算航线位置、高度、速度、姿态、油量 |
| `passenger_model` | 生成乘客、座位、终端、偏好 |
| `window_model` | 保存并增量更新舷窗状态 |
| `ife_model` | 生成乘客行为事件 |
| `payload_adapters` | 把内部对象转换为各接口 payload |
| `udp_sender` | UDP 发送 |
| `scheduler` | 按接口发送周期调度 |

## 9. 开发优先级

1. 实现配置加载、端口宏和 UDP 发送基础能力。
2. 实现场景上下文：单架飞机、随机航线、统一航班号、统一时间。
3. 实现乘客和座位模型：A330-200 的 237 名乘客、终端、座位、偏好。
4. 实现 `qar.frame` 连续飞行数据。
5. 实现 `smart_window.status` 增量变化。
6. 实现 `ife_633.behavior`，支持视频和音乐多类型偏好、分包发送。
7. 复用 IFE 逻辑实现 `ife_cockrell.behavior`。
8. 用固定或低频变化数据补齐 ground 系列接口。

## 10. 验收检查点

| 检查项 | 标准 |
| --- | --- |
| UDP 发送 | 所有接口能发到本地配置端口 |
| 端口配置 | 默认从 `8090` 递增，支持配置覆盖 |
| 发送周期 | 默认 10 秒，每个接口可单独配置 |
| 单航班一致性 | 所有接口的航班号、任务 ID、时间上下文一致 |
| QAR 合理性 | 经纬度连续，高度、速度、姿态符合巡航状态 |
| 舷窗细节 | 每次亮度小幅变化，状态有连续性 |
| IFE 偏好 | 每个用户有稳定的视频和音乐偏好，允许多类型 |
| IFE 分包 | 乘客多时支持 `page`、`pageSize`、`total`、`items` |
| 633/科克瑞尔统一 | 共用乘客和行为基础逻辑，仅在接口字段上适配差异 |
| A330-200 座位 | 237 个唯一座位，公务舱 30、经济舱 207，不产生布局外座位 |
| 智慧舷窗数量 | 116 个唯一 ID，左右各 58，每侧区域数量为 17/20/21 |
