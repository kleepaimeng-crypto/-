# 飞机轨迹历史回放产品规格

## 1. 文档信息

| 项 | 内容 |
| --- | --- |
| 文档状态 | 待评审的实施基线 |
| 页面 | 飞机轨迹回放系统 |
| 前端 | Vue 3 + TypeScript + Vite + OpenLayers |
| 后端 | Java 21 + Spring Boot + MyBatis + Flyway |
| 数据库 | PostgreSQL，现有 `cabin_data_platform` 中的 `flight_history` Schema |

本规格只授权单机历史轨迹回放。当前单机实时轨迹、乘客实时动态、数据管理和用户管理仍遵循各自已有契约。

## 2. 目标

用户可以筛选已经结束的飞行航段，读取其不可变历史 QAR 轨迹，并在地图上按真实时间播放。播放进度驱动飞行状态卡和曲线图，用户可以暂停、拖动进度和切换倍速。

成功标准：

1. 已结束会话能可靠归档到 `flight_history`，归档失败不会阻塞实时 UDP 入库。
2. 历史航段可按完成时间范围查询，列表分页稳定。
3. 回放地图、状态卡和曲线均只使用归档 QAR 点，不预测、不插值伪造轨迹。
4. 历史轨迹不进入自动删除流程。
5. 断流、帧号重置和正常落地都能留下可追溯的结束原因。

## 3. 范围

### 3.1 本期实现

- 在现有数据库创建 `flight_history` Schema 及历史会话、轨迹点、归档任务表。
- 在实时 `flight_session` 增加定时收尾：断流超时和稳定落地均可结束活动会话。
- 将 `FINISHED` 会话异步归档，支持幂等、失败记录和重试。
- 历史航段列表、时间范围筛选、分页和排序。
- 历史航段详情及指定时间范围内的轨迹点查询。
- 回放页面、地图轨迹、飞机标记、状态卡、曲线图、进度条、播放/暂停和倍速控制。
- loading、error、empty、normal 四种页面状态。

### 3.2 本期不实现

- 多架飞机同时回放、航段对比或多窗口同步回放。
- 基于历史点的插值、预测、纠偏或路线规划。
- 修改、删除、恢复、导出历史轨迹。
- 历史数据上云、异地 PostgreSQL、消息队列或 Redis。
- 新增角色、权限点或全站鉴权重构。
- 改变当前实时轨迹接口 `GET /api/v1/flight-track/current` 的语义。

## 4. 飞行会话与收尾规则

### 4.1 当前会话归属

实时 QAR 按 `source_system_code + source_device_code + source_host` 查找唯一 `ACTIVE` 会话。新 QAR 与当前会话同时满足以下条件时，归入同一会话：

- 航班号、起飞机场、目的机场相同，比较时忽略大小写。
- 接收时间与上一个点相差不超过 5 分钟。
- 采样时间与上一个点相差不超过 5 分钟。
- 帧号未从较大值重置到 `0–5`。

任一条件不满足时，旧会话结束并创建新会话。该规则沿用当前 `FlightSessionService`，本期不得改成只按航班号判断。

### 4.2 定时收尾

增加每分钟执行一次的收尾任务，扫描 `ACTIVE` 会话。任务与 QAR 入库使用同一来源流锁，并使用 `status='ACTIVE'` 条件更新，避免并发重复结束。

| 结果 | 判定规则 | 结束时间 |
| --- | --- | --- |
| `LANDED` | 最近连续 5 分钟 QAR 均满足 `air_ground_status=GROUND`、`ground_speed_kt <= 40`、`altitude_ft <= 200`，且相邻采样间隔均不超过 60 秒 | 最后一条 QAR 的 `sample_at` |
| `TIMEOUT` | `last_received_at` 距当前时间超过 5 分钟 | 最后一条 QAR 的 `sample_at` |
| `NEW_FLIGHT` | 新 QAR 的航班号或航线改变，或采样/接收时间间隔超过 5 分钟 | 旧会话最后一条 QAR 的 `sample_at` |
| `FRAME_RESET` | 新帧号重置到 `0–5` 且小于旧帧号 | 旧会话最后一条 QAR 的 `sample_at` |

`AIR GND ON GND` 是主判断信号；速度和高度只用于防止瞬态值误判。不得只用地速低于阈值判定落地。

实时表 `flight_session.status` 保持 `ACTIVE / FINISHED`，结束原因写入历史归档会话，不为实时表扩大状态枚举。

## 5. 历史归档

- 只有 `FINISHED` 会话可创建归档任务。
- 归档复制会话元数据和其全部 QAR 点到 `flight_history` Schema；不复制原始 UDP payload、`data_record`、乘客或流量数据。
- `source_flight_session_id` 在历史会话表中唯一，`source_qar_sample_id` 在历史点表中唯一；重复投递不得产生重复数据。
- 同一归档任务内，历史会话和轨迹点必须在一个数据库事务中写入。
- 归档失败记录错误摘要和重试次数；重试采用受控退避，连续失败仍保留任务供人工重试。
- 归档成功后不删除或修改 `public.flight_session`、`public.qar_sample`。
- `flight_history` 中的会话和点不执行平台自动删除任务；任何删除需求必须后续单独立项。

## 6. 历史航段查询

默认按 `endedAt DESC, id DESC` 查询。用户可按以下条件筛选：

- 航段完成时间范围，必填至少一端，最大跨度 31 天。
- 航班号、起飞机场、目的机场、飞机注册号，可选精确或前缀匹配。
- 结束原因，可选。

列表默认每页 20 条，可选 20、50、100 条。每项展示航班号、航线、飞机、开始/结束时间、时长、轨迹点数、结束原因和归档时间。

## 7. 回放交互

### 7.1 页面结构

- 顶部沿用现有平台导航，“飞机轨迹回放系统”变为可点击入口，路由为 `/flight-history`。
- 左侧为历史航段筛选及列表；中部为 OpenLayers 地图；右侧为飞行状态卡和曲线图。
- 选中航段后读取其轨迹，初始进度为首个轨迹点。

### 7.2 播放规则

- 控件：播放、暂停、回到起点、进度拖动、0.5×、1×、2×、4×、8×。
- 播放时按照相邻 QAR 点的真实 `sampleAt` 时间差推进；倍速只改变客户端播放速度。
- 进度对应的“当前点”更新地图飞机位置、状态卡和图表高亮；地图只绘制从首点到当前点的已飞轨迹。
- 曲线图使用已读历史点；当前进度用高亮游标表示，不重新请求下一页或生成虚拟点。
- 到达最后一点后自动暂停；重新播放从当前点继续，回到起点后可再次播放。
- 页面切换、重新选择航段或组件卸载时停止定时器和动画。

### 7.3 数据与异常状态

- 初始加载失败显示重试入口；后台请求失败保留上一次成功数据并提示。
- 无历史航段时展示空态，不显示假轨迹。
- 轨迹点缺失经纬度时不在地图绘制，但保留其时间顺序；无可绘制点时提示轨迹不可用。
- 后端返回的点按 `sampleAt, frameCount, sourceQarSampleId` 升序；前端不得自行改变会话边界。

## 8. 权限与安全

- 历史轨迹查询沿用当前实时轨迹访问边界：所有 `ACTIVE` 角色可读取。
- 用户状态被改为非 `ACTIVE` 后，JWT 下一次请求即被拒绝。
- 历史 API 全部只读；不提供删除、修改、下载和归档任务管理接口给前端。
- 不向历史响应返回原始 UDP payload、数据库凭据或内部错误栈。

## 9. 验收标准

1. 正常落地、断流、航班切换和帧号重置均能结束会话并记录正确结束原因。
2. 收尾任务与 QAR 入库并发时，不会重复结束或遗漏新点。
3. 历史归档可重试、可恢复，且同一源会话和源 QAR 点不重复。
4. 历史表不受实时数据 7+7 天保留任务影响。
5. 历史航段列表筛选、分页、排序和总数正确。
6. 回放地图、状态卡和曲线图均以真实归档点驱动，支持拖动、暂停和五种倍速。
7. 1920×1080 与 1366 宽度下可完成筛选和回放核心流程。
8. 后端测试、前端类型检查、前端测试及生产构建通过。
