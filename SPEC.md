# 前中后舱网联数据显示平台需求规格（首期）

## 1. 文档信息

| 项 | 内容 |
| --- | --- |
| 文档状态 | 首期开发基线 |
| 首期范围 | 数据管理、UDP 接入、查询详情、标签批注、软删除、导入导出 |
| 技术基线 | Vue 3 + TypeScript + Element Plus；Java 21 + Spring Boot；PostgreSQL 18 |
| 数据来源 | 项目需求 Word、6 张界面预设图、`simulator/` 下的接口 Schema 与模拟器代码 |
| 非首期页面 | 飞机轨迹实时、轨迹回放、数据统计、用户管理、乘客实时动态 |

本文档是首期产品范围的唯一需求基线。`API.md` 和 `schema.md` 必须与本文档保持一致；未写入本文档的功能不能由开发人员或 Agent 自行扩展。

## 2. 项目目标

平台首期要形成一条可以验收的真实数据链路：

```text
7 路 UDP 模拟数据 -> Spring Boot 接收与解析 -> PostgreSQL 保存
-> REST API -> Vue 数据管理页 -> 查询、管理与文件导出
```

首期成功的标准不是只还原一张静态界面，而是管理员能够登录系统，启动现有模拟器后看到真实入库的数据，并完成筛选、查看详情、标签、批注、软删除、恢复、CSV 导入和 CSV/PDF 导出。

## 3. 用户与权限

首期只有一种用户：平台管理员。

管理员可以：

- 登录和退出平台。
- 查询全部可见数据。
- 查看只读原始报文和解析摘要。
- 添加、修改、删除标签与批注。
- 批量添加标签、批注和软删除数据。
- 恢复尚在恢复期内的数据。
- 下载 CSV 模板、导入 CSV、查看导入历史与错误报告。
- 按筛选条件导出 CSV/PDF，并查看导出历史。

首期不实现用户列表、角色配置、权限点配置和多角色 RBAC。数据库仍保留管理员状态字段，为后续扩展留出兼容空间。

## 4. 数据来源

### 4.1 UDP 通道

| 端口 | 消息类型 | 来源设备编码 | 首期处理方式 |
| ---: | --- | --- | --- |
| 8090 | `qar.frame` | `SIM-QAR` | 保存完整 JSONB，并解析一条 `qar_sample` |
| 8091 | `ground.task` | `SIM-GROUND` | 保存完整 JSONB，并解析一条 `simulation_task` 快照 |
| 8092 | `ground.traffic_record` | `SIM-GROUND` | 一份批量报文保存为一条目录记录，`items` 拆入 `traffic_record` |
| 8093 | `ground.session_summary` | `SIM-GROUND` | 一份批量报文保存为一条目录记录，`items` 拆入 `session_summary` |
| 8094 | `smart_window.status` | `SIM-WINDOW` | 一份批量报文保存为一条目录记录，`items` 拆入 `smart_window_status` |
| 8095 | `ife_633.behavior` | `SIM-IFE-633` | 每个分页报文保存为一条目录记录，乘客行为拆入 `ife_633_behavior` |
| 8096 | `ife_cockrell.behavior` | `SIM-IFE-COCKRELL` | 每个分页报文保存为一条目录记录，乘客行为拆入 `ife_cockrell_behavior` |

一份 UDP 数据报在统一数据目录中对应一条 `data_record`。批量报文中的 `items` 会拆入对应业务表，但不能把 320 个乘客行为拆成 320 条目录记录；它们仍通过同一个 `record_id` 追溯到各自分页原报文。

模拟器不需要提供 `messageId`。平台在接收时生成 UUID 作为记录主键。首期把每次收到的数据报视为一次真实传输，不做基于载荷内容的强制去重。

### 4.2 后端补齐的管理字段

| 管理字段 | 来源 |
| --- | --- |
| 数据记录 ID | 后端生成 UUID |
| 数据类型 | UDP 监听端口映射 |
| 来源系统与来源设备 | UDP 监听配置映射 |
| 接收时间 | 后端实际收包时间 |
| 发送时间 | 优先使用报文 `sentAt`；QAR 用接收日期与 `time` 合并 |
| 飞机注册号、机型 | 开发配置补齐测试飞机 `B-TEST-001` / `Boeing 777-300ER` |
| 航班与航段 | 从 QAR、任务或 IFE 解析；舷窗关联当前活动航班 |
| 航司 | 从航班号或接收配置提取并直接保存到数据目录，测试数据为 `CA` |
| 标签与批注 | 管理员在平台维护 |

“设备”筛选指报文来源设备，不指乘客座椅终端。终端 ID、座位号等首期保留在原始 JSON 中。

## 5. 功能需求

### 5.1 登录

- 管理员使用用户名和密码登录。
- 登录成功后获得 JWT；未登录访问业务接口返回 401。
- 禁用用户不能登录。
- 原始密码不得写入日志、数据库明文或代码仓库。

### 5.2 数据列表

列表支持以下筛选条件：

- 标签，可多选。
- 航司代码。
- 航班号。
- 来源设备。
- 机型。
- 起飞机场和到达机场。
- 数据类型。
- 接收时间范围。

列表默认按接收时间倒序，支持按数据类型、发送时间、接收时间排序。采用服务端分页，默认每页 20 条，可选 20、50、100 条。

表格至少展示：选择框、飞机注册号、数据类型、发送时间、接收时间、解析状态和操作。页面默认每 5 秒刷新一次，管理员可以暂停自动刷新；编辑、选择数据或切换非第一页时不得强制跳页。

### 5.3 数据详情

点击记录打开详情抽屉，包含：

- 飞机、航班、来源设备、数据类型和时间等管理信息。
- 格式化后的原始 JSON；解析失败时展示保存的原始文本。
- 从原始 JSON 动态生成的公共字段和轻量解析摘要。
- 当前标签和批注历史。
- 解析状态与错误原因。

原始报文和解析摘要只读。管理员只能修改标签、批注及允许编辑的管理元数据，不得覆盖原始报文。

### 5.4 标签与批注

- 标签名称全局唯一，支持名称、颜色和启用状态。
- 一条数据可以关联多个标签，一个标签可以关联多条数据。
- 支持对所选记录批量添加或移除标签。
- 一条数据可以有多条批注；批注记录创建人、创建时间、修改时间。
- 支持单条和批量新增批注；删除批注使用软删除。

### 5.5 删除与恢复

- 数据删除采用软删除，列表默认不显示已删除数据。
- 删除必须二次确认并记录审计日志。
- 管理员可以在恢复期内恢复记录。
- 原始数据正常展示 7 天；超过 7 天由系统标记软删除，再保留 7 天恢复期，之后批量物理清理。
- 物理清理可以级联清除数据标签关系和批注；审计日志必须保留目标 ID 与操作快照。

### 5.6 CSV 导入

首期提供四种模板：QAR、任务、流量、会话。

- 一次导入只能选择一种数据类型。
- 单文件最大 50 MB，最多 100,000 行。
- 校验表头、必填字段、时间、数值、枚举和长度。
- 分批写入数据库；单个坏行不能导致全部成功行回滚。
- 导入任务记录总行数、成功数、失败数和状态。
- 存在失败行时生成可下载的错误 CSV，包含原行和失败原因。
- 导入数据与 UDP 数据使用相同的 `data_record + 对应业务表` 模型，并保留原始 JSONB。

### 5.7 CSV/PDF 导出

- 导出使用创建任务时的筛选条件快照，不能因页面后续改变而改变结果。
- CSV 最多导出 100,000 行；PDF 最多导出 5,000 行，超过时提示使用 CSV。
- 导出异步执行，记录创建人、格式、数据类型、状态、行数、文件和时间。
- 导出文件只包含当前管理员有权访问且未删除的数据。
- 导入历史和导出历史分别展示，修正界面预设图中两者标题与时间列互换的问题。

### 5.8 审计

以下动作必须写入只追加的审计日志：登录成功/失败、修改管理元数据、标签变更、批注变更、删除、恢复、创建导入任务、创建导出任务、下载导出文件。

审计记录至少包含操作、目标类型、目标 ID、操作者、时间、请求 IP、操作前值和操作后值。

## 6. 非功能要求

### 6.1 性能与容量

- 默认模拟器频率下，7 路 UDP 持续接收不得因正常载荷大小丢包。
- 100,000 条目录记录下，常用筛选的第一页查询目标为 1 秒内完成。
- 列表和导出禁止一次把全部结果加载到 JVM 内存。
- 原始 JSON 不建立全字段 GIN 索引；首期只索引明确的管理字段和业务查询字段。

### 6.2 可追溯性

- 原始报文不可更新。
- 所有时间使用 PostgreSQL `timestamptz`，前端按 `Asia/Shanghai` 展示。
- 数据库结构只能通过 Flyway 迁移变更。
- 导入、UDP 接收、管理员修改均能追溯到来源和时间。

### 6.3 兼容性与运行环境

- 数据库由 Docker Compose 启动 PostgreSQL 18，并使用命名卷持久化。
- 后端使用 Java 21、Spring Boot 4.1.x、Maven。
- 前端使用 Vue 3、TypeScript、Vite 8、Element Plus，开发机使用 Node 22 LTS。
- 首期桌面端目标分辨率为 1920x1080，最低支持 1366 像素宽度。

## 7. 首期验收标准

1. Docker Compose 能启动健康的 PostgreSQL，后端启动时 Flyway 自动完成迁移和种子数据初始化。
2. 管理员可以登录，未登录或禁用账号不能访问数据接口。
3. 启动现有模拟器后，七种消息均能生成目录记录；完整原报文和公共管理字段正确入表。
4. 飞机、航班、来源设备、数据类型、发送时间和接收时间映射正确。
5. 筛选、排序、分页、自动刷新和详情抽屉可正常使用。
6. 原始报文只读；标签、批注、软删除、恢复和审计可正常使用。
7. 四种 CSV 模板可以下载并导入；错误文件可下载且原因明确。
8. CSV/PDF 能按筛选快照导出，任务状态与历史记录准确。
9. 非法 JSON、缺少字段、非法时间、文件超限和数据库错误均返回明确错误，不泄露堆栈或密码。
10. 后端测试、数据库集成测试、前端类型检查、测试和生产构建全部通过。
