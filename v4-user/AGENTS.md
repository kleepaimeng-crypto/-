# 用户管理实施规则

## 1. 适用范围

本文件约束用户管理功能的文档、数据库、后端和前端实现。本目录只覆盖基础契约中“用户管理暂不开发”的限制，不覆盖其他页面和模块的范围限制。

## 2. 必读顺序

开发前按顺序阅读：

1. `v4-user/SPEC.md`
2. `v4-user/API.md`
3. `v4-user/schema.md`
4. `v4-user/SLICES.md`
5. `v4-user/AGENTS.md`
6. `v1-docs` 基础契约和当前相关源码

如本文档集内部冲突，优先级为 `SPEC.md`、`API.md`、`schema.md`、`SLICES.md`、`AGENTS.md`，并在实现前先修正文档。

## 3. 范围约束

只实现：

- 用户分页列表和排序。
- 添加、编辑、软删除。
- 三角色和四状态。
- 用户管理专属访问控制。
- 用户写操作审计。
- 参考图对应的用户管理页面。

禁止顺手实现：

- 完整 RBAC。
- 权限点、角色、部门或组织配置。
- 批量用户操作。
- 邮件激活、密码找回或密码重置。
- 全站业务接口权限重构。
- 与用户管理无关的前后端重构。

## 4. 代码原则

- 遵循 `andrej-karpathy-skills:karpathy-guidelines`。
- 先契约、后迁移、再后端、最后前端。
- 修改范围保持最小，每行修改都应能追溯到用户管理需求。
- 不抽象一次性框架，不引入新状态管理库或权限库。
- 匹配当前项目 Java、Vue、MyBatis 和测试风格。
- 只清理本次修改产生的未使用代码。
- 保留工作区已有用户改动。

## 5. 数据库规则

- 只能通过新 Flyway `V10` 修改 Schema。
- 禁止修改已发布迁移。
- 禁止物理删除 `app_user`。
- 禁止删除或重建用户历史外键。
- 删除必须写 `DELETED`、删除时间、删除人、删除原因。
- 用户名和邮箱继续保持不区分大小写唯一，包括已删除用户。
- 编辑和删除必须使用 `version` 乐观锁。
- 最后超级管理员校验必须防止并发竞态。

## 6. 后端规则

- 用户管理代码放入独立 `com.cabin.user` 包，不塞进 `login`、`data` 或 `passenger`。
- Controller 只处理协议、校验和当前用户；Service 处理事务和业务规则；Mapper 处理 SQL。
- `/api/v1/users/**` 必须由后端限制为 `ROLE_SUPER_ADMIN`。
- 排序字段使用白名单映射，不把客户端字符串直接拼入 SQL。
- Entity/Row 不直接作为 API 响应。
- 所有写操作必须审计。
- 密码只在创建请求到密码编码器之间存在，不回显、不写日志、不写审计。
- 唯一约束和版本冲突转换为稳定业务错误，不返回 SQL 或数据库约束名。
- 当前用户和最后一个可用超级管理员必须受到保护。

## 7. 前端规则

- 使用 Vue 3 Composition API 和 `<script setup lang="ts">`。
- API 调用集中在 `frontend/src/api/users.ts`。
- DTO、角色、状态、查询和表单使用显式 TypeScript 类型，禁止 `any`。
- 分页和排序由后端执行。
- 页面处理 loading、error、empty、normal。
- 提交时防止重复操作；失败时保留用户输入。
- 密码和确认密码不得保存到全局状态、sessionStorage 或 localStorage。
- 只有超级管理员显示可点击的用户管理入口。
- 前端路由守卫不是安全边界，接口仍必须处理 403。
- 参考图用于布局和信息层级，不写固定用户假数据。
- 不实现没有业务动作的复选框。

## 8. 安全规则

- 只有 `ACTIVE` 用户可以登录和继续使用 JWT。
- `PENDING`、`FROZEN`、`DELETED` 统一拒绝认证。
- 超级管理员权限从数据库当前用户记录生成，不只相信 JWT 中的旧角色声明。
- 当前用户不能删除、冻结或降级自己。
- 系统始终保留至少一个 `ACTIVE SUPER_ADMIN`。
- 密码和密码哈希不得出现在响应、日志、异常和审计快照。
- 删除用户不可恢复，实施前端必须明确提示。

## 9. 验证要求

至少运行：

```powershell
Set-Location backend
.\mvnw.cmd test
.\mvnw.cmd verify

Set-Location ..\frontend
npm run typecheck
npm run test
npm run build
```

并验证：

- V9 到 V10 迁移和空库迁移。
- 三种角色的接口权限。
- 四种状态的登录行为。
- 唯一冲突、乐观锁和最后超级管理员保护。
- 软删除和历史外键。
- 审计脱敏。
- 1920×1080、1366 宽度页面布局。

## 10. 禁止事项

- 禁止批量删除文件或目录。
- 禁止使用 `del /s`、`rd /s`、`rmdir /s`、`Remove-Item -Recurse`、`rm -rf`。
- 禁止修改模拟器。
- 禁止升级依赖。
- 禁止格式化整个仓库。
- 禁止修改与用户管理无关的飞行轨迹、乘客实时动态、数据管理或部署逻辑。
- 禁止为了展示页面而返回固定用户假数据。

## 11. 完成汇报

实现完成后必须说明：

- 用户可见行为。
- 修改的契约、迁移、后端和前端文件。
- 数据迁移和兼容结果。
- 验证命令及结果。
- 权限、并发和审计覆盖情况。
- 工作区是否包含无关改动。
