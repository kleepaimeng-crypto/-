# 用户管理产品规格

## 1. 文档信息

| 项 | 内容 |
| --- | --- |
| 文档状态 | 待评审的实施基线 |
| 页面 | 用户管理 |
| 视觉参考 | `docs/前中后舱网联数据平台 5.png` |
| 前端 | Vue 3 + TypeScript + Vite + Element Plus |
| 后端 | Java 21 + Spring Boot + MyBatis + Flyway |
| 数据库 | PostgreSQL |

本规格只授权用户管理垂直切片。基础登录、数据管理、乘客实时动态和飞机实时轨迹继续遵循各自现有契约。

## 2. 目标

超级管理员可以在平台内完成用户查询、添加、编辑和删除，并且角色、状态变化立即影响用户管理权限和登录资格。所有写操作必须可审计，不得泄露密码。

成功标准：

1. 用户列表来自真实后端接口，支持服务端分页和排序。
2. 超级管理员可以添加、编辑和软删除用户。
3. 普通管理员和普通用户不能访问页面或调用用户管理接口。
4. 非 `ACTIVE` 用户不能登录，已登录用户在状态变更后下一次请求立即失效。
5. 不能删除、冻结或降级当前登录用户，系统始终至少保留一个可用超级管理员。
6. 数据库迁移保留现有用户和历史外键关系。

## 3. 范围

### 3.1 本期实现

- 用户管理导航入口和独立路由。
- 用户分页列表。
- 用户名、角色、邮箱、状态列排序。
- 添加用户。
- 编辑用户名、邮箱、角色和状态。
- 删除用户，采用终态软删除。
- loading、error、empty、normal 四种页面状态。
- 添加、编辑、删除的成功反馈和失败提示。
- 服务端权限校验、输入校验、乐观锁和审计。

### 3.2 本期不实现

- 角色或权限点配置页面。
- 完整 RBAC 表结构。
- 完整 RBAC、权限点配置或角色配置页面。
- 用户详情页。
- 批量添加、批量编辑、批量删除。
- 邮件邀请、邮件激活、验证码。
- 密码找回、密码重置、用户自行修改密码。
- 登录会话列表、强制下线接口。
- 用户物理删除或恢复已删除用户。
- 用户头像、手机号、部门、职位等参考图未要求的字段。

## 4. 角色模型

| 角色代码 | 页面文案 | 本期权限 |
| --- | --- | --- |
| `SUPER_ADMIN` | 超级管理员 | 访问现有业务页面；访问用户管理；添加、编辑、删除用户；编辑和删除数据管理记录 |
| `ADMIN` | 管理员 | 访问现有业务页面；不能访问用户管理；可以编辑和删除数据管理记录 |
| `USER` | 普通用户 | 访问现有业务页面；不能访问用户管理；数据管理页面只读，编辑、删除和批量删除按钮置灰不可点 |

本期不实现完整 RBAC；仅补充数据管理记录写操作边界。数据管理记录编辑、删除、批量删除、恢复和标签批量变更接口只允许 `SUPER_ADMIN`、`ADMIN` 调用，`USER` 调用返回 403 `ACCESS_DENIED`。数据查询、飞机实时轨迹和乘客实时动态继续允许三种角色访问。

只有后端权限校验构成安全边界。前端隐藏导航和路由拦截只用于改善体验，不能代替后端授权。

## 5. 状态模型

| 状态代码 | 页面文案 | 可以登录 | 可以编辑 | 可以删除 |
| --- | --- | ---: | ---: | ---: |
| `ACTIVE` | 激活 | 是 | 是 | 是 |
| `PENDING` | 未激活 | 否 | 是 | 是 |
| `FROZEN` | 冻结 | 否 | 是 | 是 |
| `DELETED` | 删除 | 否 | 否 | 否 |

状态规则：

- 新建用户默认 `PENDING`，超级管理员也可以在创建时明确选择 `ACTIVE`。
- `PENDING` 的激活由超级管理员在编辑操作中完成，本期没有邮件激活流程。
- `FROZEN` 可以由超级管理员恢复为 `ACTIVE` 或改为 `PENDING`。
- `DELETED` 是不可恢复终态，列表继续展示该用户，但编辑和删除按钮禁用。
- 登录和 JWT 认证只接受数据库当前状态为 `ACTIVE` 的用户。
- 状态改变后，JWT 过滤器在下一次请求回查数据库时立即执行最新状态。

## 6. 用户字段与校验

| 字段 | 规则 |
| --- | --- |
| 用户名 | 必填；trim 后转小写；3～64 个字符；只允许字母、数字、点、下划线和连字符；不区分大小写唯一 |
| 邮箱 | 新建用户必填；trim 后转小写；合法邮箱；最长 254；不区分大小写唯一 |
| 初始密码 | 新建用户必填；6～72 个字符；仅用于 BCrypt 哈希，不回显、不记录日志或审计 |
| 角色 | 必须为 `SUPER_ADMIN`、`ADMIN`、`USER` |
| 状态 | 新建时只能为 `ACTIVE` 或 `PENDING`；编辑时允许 `ACTIVE`、`PENDING`、`FROZEN` |
| 版本 | 编辑和删除必须携带 `expectedVersion`，用于乐观锁 |
| 删除原因 | 删除必填；trim 后 1～500 个字符 |

已存在但邮箱为空的历史账号允许继续存在；首次编辑时必须补齐合法邮箱。

不符合新用户名格式的历史账号允许继续登录和展示；首次编辑用户名时必须满足新格式。迁移不得为了满足新格式自动改写历史登录名。

已删除用户名和邮箱继续占用唯一值，不允许重新注册同名或同邮箱用户，避免审计身份混淆。

## 7. 用户列表

### 7.1 展示

表格展示：

- 序号：由当前页和页大小计算，不作为数据库字段。
- 用户名。
- 权限。
- 邮箱。
- 状态。
- 操作：编辑、删除。

列表默认每页 20 条，可选 20、50、100 条。默认按创建时间倒序；支持按用户名、角色、邮箱、状态和创建时间排序。服务端分页和排序，前端不得加载全量用户后处理。

列表包含 `DELETED` 用户，以便与参考图和审计追溯一致。删除用户的操作按钮禁用。

参考图中的复选框没有对应业务动作，本期不展示复选框，也不提供批量接口。

### 7.2 页面状态

- initial loading：首次加载显示明确加载状态。
- error：首次加载失败显示错误和重试入口。
- empty：没有用户时显示空态；正常部署至少应有一个超级管理员。
- normal：显示表格和分页。
- background error：写操作后重新加载失败时保留当前列表，并提示刷新失败。

## 8. 添加用户

点击“用户添加”打开对话框，字段包括用户名、邮箱、初始密码、确认密码、角色和状态。

交互规则：

- 前端做格式提示，后端执行最终校验。
- 两次密码输入必须一致；确认密码只存在前端，不发送到后端。
- 默认角色为 `USER`，默认状态为 `PENDING`。
- 提交期间禁用重复提交。
- 添加成功后关闭对话框，回到第一页并重新加载列表。
- 用户名或邮箱冲突时，在对应字段附近显示后端返回的冲突信息。

## 9. 编辑用户

点击“编辑”打开对话框，显示用户名、邮箱、角色和状态，不展示密码字段。

编辑规则：

- 必须携带当前行 `version`。
- 当前登录用户不能修改自己的角色，也不能把自己的状态改为非 `ACTIVE`。
- 已删除用户不能编辑。
- 将超级管理员改为其他角色或非 `ACTIVE` 状态前，后端必须确认仍有另一个 `ACTIVE SUPER_ADMIN`。
- 编辑成功后更新列表；版本冲突时提示用户刷新后重试。

## 10. 删除用户

点击“删除”打开二次确认对话框，显示用户名并要求填写删除原因。

删除规则：

- 删除为把状态改为 `DELETED`，同时记录删除时间、删除人、删除原因并增加版本号。
- 当前登录用户不能删除自己。
- 已删除用户重复删除返回冲突。
- 删除 `ACTIVE SUPER_ADMIN` 前必须确认仍有另一个 `ACTIVE SUPER_ADMIN`。
- 删除不移除用户行，不修改审计、标签、批注、数据记录或文件任务中的历史外键。
- 删除成功后保留当前页；若当前页因删除导致超出总页数，则退到最后一页。

## 11. 导航与访问控制

- 新增路由 `/users`，要求已登录且当前角色为 `SUPER_ADMIN`。
- 超级管理员在数据管理、飞机实时轨迹、乘客实时动态和用户管理页面看到可点击的“用户管理”导航。
- `ADMIN` 和 `USER` 不显示可点击入口。
- 非超级管理员直接访问 `/users` 时跳转到默认工作区并显示无权限提示。
- 非超级管理员直接调用 `/api/v1/users/**` 返回 403 `ACCESS_DENIED`。
- 用户被删除或冻结后，现有 JWT 在下一次请求时返回 403 `ACCOUNT_DISABLED` 并由前端清理会话。

## 12. 审计和安全

以下操作写入 `audit_log`：

- `CREATE_USER`
- `UPDATE_USER`
- `DELETE_USER`

审计要求：

- `target_type` 为 `APP_USER`，`target_id` 为用户 UUID。
- 保存操作者、请求 IP、traceId、结果和脱敏前后快照。
- 快照可以包含用户名、邮箱、角色、状态和版本。
- 密码、密码哈希、JWT、确认密码不得进入响应、日志或审计。
- 用户名和邮箱冲突错误不得泄露数据库约束名或 SQL。

## 13. 错误处理

| 场景 | HTTP | 错误码 |
| --- | ---: | --- |
| 未登录或 JWT 无效 | 401 | `UNAUTHORIZED` |
| 已登录但不是超级管理员 | 403 | `ACCESS_DENIED` |
| 当前账号不是 ACTIVE | 403 | `ACCOUNT_DISABLED` |
| 字段或排序参数非法 | 400 | `VALIDATION_ERROR` |
| 用户不存在 | 404 | `RESOURCE_NOT_FOUND` |
| 用户名/邮箱重复、版本冲突、自我锁定、最后超级管理员保护 | 409 | `RESOURCE_CONFLICT` |
| 重复删除 | 409 | `RESOURCE_CONFLICT` |
| 数据库不可用 | 503 | `DATABASE_UNAVAILABLE` |

错误响应继续使用现有统一响应和 traceId，不返回堆栈、SQL 或约束名。

## 14. 验收标准

1. 现有数据库可通过新 Flyway 迁移升级，历史用户和外键记录不丢失。
2. 现有 `ADMIN` 用户升级为 `SUPER_ADMIN`，现有 `DISABLED` 用户升级为 `FROZEN`。
3. 超级管理员可以访问 `/users`，其他角色不能访问页面或接口。
4. 用户列表使用真实 API，分页、页大小、排序和总数准确。
5. 添加用户完成校验、密码哈希、唯一冲突处理和审计。
6. 编辑用户完成乐观锁、自我保护、最后超级管理员保护和审计。
7. 删除用户为不可恢复软删除，历史外键和审计仍可查询。
8. `PENDING`、`FROZEN`、`DELETED` 用户不能登录。
9. 页面正确处理 loading、error、empty 和 normal。
10. 1920×1080 下与参考图保持相同信息层级，1366 像素宽度下仍可完成核心操作。
11. 后端测试、前端类型检查、前端测试和生产构建通过。
12. 本期没有新增角色表、权限表、邮件流程、密码重置或无关模块重构。
